# 本地投屏最近观看设计

## 背景

“最近观看”页面当前只调用 B 站历史接口，投屏播放则依赖播放器心跳让记录最终进入账号历史。该机制受登录状态、接口可用性和服务端同步延迟影响。为让用户能立即找回在本机成功播放过的投屏视频，需要增加独立的本地投屏最近观看数据，并与原有 B 站历史合并展示。

本设计扩展 `2026-07-12-cast-title-history-design.md` 中“最近观看来源仍为 B 站历史接口，不新增本地投屏最近观看列表”的原范围。B 站历史仍是账号历史来源；新增数据只记录本机投屏观看，不替代或冒充账号历史。

## 目标

- 投屏普通视频首次触发媒体 `playing` 状态后，将可重播的视频写入本地最近观看。
- “最近观看”页面继续请求 B 站历史，并把本地投屏记录合并到同一个列表。
- 同一视频跨来源只展示一次，按两条记录中较新的观看时间排序。
- 未登录、B 站历史接口失败或超时时，仍展示可用的本地记录，并明确提示账号历史未加载。
- 本地数据损坏或存储失败不能影响视频播放或 B 站历史展示。

## 非目标

- 不以本地列表替代 B 站账号历史，也不改变 `getHistory` 接口及其认证方式。
- 不复用 `resume_progress` 作为最近观看数据源。
- 不持久化缺少 `bvid`、无法通过现有卡片流程重播的投屏内容。
- 不在 webOS 后台服务中增加历史存储或 Luna 查询接口。
- 不为普通的应用内播放写入这份投屏专用历史。
- 不新增来源标签、独立分区或页内标签切换。

## 推荐方案

在前端增加独立、版本化的本地投屏历史存储。`PlayerPage` 仅在 `fromCast: true` 的普通视频首次触发媒体 `playing` 状态且存在 `bvid` 后写入记录。`HistoryPage` 并行读取本地记录与 B 站历史，将两种数据归一化、合并、按 `bvid` 去重并按时间倒序交给现有 `VideoGrid`。

该方案让记录时机与实际播放一致，复用现有前端存储容错方式，不要求后台服务增加状态和接口，也能在浏览器开发环境与电视环境中保持同一行为。

## 数据模型与存储规则

本地记录应使用独立于 `resume_progress` 的存储键。存储值包含 schema 版本和记录数组，以便未来迁移与容错。单条记录的概念字段如下：

| 字段 | 必需 | 含义 |
| --- | --- | --- |
| `bvid` | 是 | 去重键及现有卡片重播所需标识 |
| `cid` | 否 | 最近一次投屏的分 P 标识 |
| `title` | 否 | 已解析的投屏命令或视频详情标题 |
| `pic` | 否 | 视频封面 |
| `ownerName` | 否 | UP 主名称 |
| `duration` | 否 | 视频总时长，单位沿用现有卡片模型 |
| `progress` | 否 | 最近有效播放进度 |
| `viewedAt` | 是 | 首次进入媒体 `playing` 状态的毫秒时间戳 |

读取时应归一化数据：

- 丢弃没有合法 `bvid` 或 `viewedAt` 的记录。
- 可选字段类型不合法时仅移除该字段，不丢弃整条可重播记录。
- 损坏、未知版本或非预期顶层结构视为空列表。
- 存储读取异常视为空列表，不向页面抛出。

写入时应：

1. 读取并归一化现有记录。
2. 按 `bvid` 删除旧记录。
3. 把本次记录放到最前，保留本次已知元数据；若本次缺少某个展示字段，可沿用旧记录中的有效值。
4. 按 `viewedAt` 倒序排列并裁剪为最多 50 条。
5. 吞掉存储配额或平台异常，不能中断播放。

同一 `bvid` 的不同分 P 在最近观看中只占一项。再次投屏时，`cid` 和进度更新为最近一次观看的数据。

## 写入时机

本地历史写入必须满足以下全部条件：

- 当前内容为普通视频投屏，即 `fromCast: true`，而非直播或应用内普通播放。
- 当前播放会话首次触发媒体 `playing` 状态，而不只是收到投屏命令、完成页面导航或加载完元数据。
- 视频存在非空 `bvid`。

每个投屏播放会话只在第一次 `playing` 事件写入一次。同一会话因缓冲恢复、暂停后继续或媒体事件重复触发时不得再次写入，也不得刷新 `viewedAt`，否则长时间播放会持续占据“最新”位置。本地历史保存首次播放时的进度；后续细粒度进度仍由现有 `resume_progress` 负责，两个数据集保持职责分离。用户稍后重新发起同一视频的投屏属于新会话，应更新该视频的本地记录和 `viewedAt`。

播放器应使用当时已解析出的标题、封面、作者、时长和进度写入。元数据请求失败不阻止记录，只要 `bvid` 可重播；以后与 B 站记录合并时可补全缺失字段。

## 合并与去重

`HistoryPage` 进入或刷新时并行获取：

1. `getHistory(0, 0, 24)` 返回的 B 站账号历史。
2. 本地投屏最近观看记录。

两类数据先转换为统一的历史展示模型，再按 `bvid` 合并。合并规则如下：

- 每个 `bvid` 最多产生一张卡片。
- 排序前将 B 站历史观看秒级时间转换为毫秒；排序时间取转换后的 B 站时间与本地 `viewedAt` 中的较新值。
- `cid` 和 `progress` 取观看时间较新的来源；若较新来源缺少有效值，则回退到另一来源。
- 标题、封面、作者、时长等展示字段优先取较新来源的有效值，再由另一来源补全。
- 两条 B 站记录的时间相同或缺少可用时间时，保留接口返回顺序；本地记录自身使用 `viewedAt` 保证稳定排序。
- 不跨不同 `bvid` 去重，也不以标题作为去重依据。

B 站历史仍只请求现有的 24 条；本地最多 50 条，因此合并列表最多出现 74 个不同 `bvid`。本设计不增加分页。

## 页面状态与刷新

合并结果继续使用现有 `VideoGrid`，不增加来源分区或标签。

- **B 站历史成功**：展示合并、去重、排序后的列表。
- **B 站历史未登录、报错或超时，且本地有记录**：展示本地列表，同时显示非阻塞提示，明确说明 B 站历史未加载及原因；接口错误应保留现有重试能力。
- **B 站历史不可用，且本地为空**：保留当前登录提示或错误/重试状态。
- **B 站历史为空，且本地有记录**：正常展示本地记录，不显示空态。
- **两边均为空**：显示现有空态。

重新进入“最近观看”，以及在该页面已激活时重新选择对应侧栏项，都应重新读取本地记录并请求 B 站历史。这样刚完成的投屏记录无需重启应用即可出现。

## 数据流

```text
投屏服务发布普通视频命令
  -> App 构造 fromCast 播放对象
  -> PlayerPage 解析视频、标题和播放资源
  -> 播放器成功开始播放
  -> bvid 存在时写入本地投屏最近观看（最多 50 条）

用户打开或刷新“最近观看”
  -> 并行读取本地投屏记录与请求 B 站历史
  -> 分别归一化
  -> 按 bvid 合并并补全字段
  -> 按最新观看时间倒序
  -> VideoGrid 展示单一列表
```

## 错误处理

- 本地存储读取、解析或写入失败不得中断播放器，也不得遮蔽成功返回的 B 站历史。
- 单条本地记录损坏时丢弃该条，其他合法记录继续使用。
- B 站历史失败不得被伪装成完整成功状态；只要展示本地降级结果，就必须同时显示账号历史未加载提示。
- 投屏启动失败不得写入本地最近观看。
- 缺少 `bvid` 的投屏即使能够临时播放也不写入，避免生成无法通过现有卡片流程重播的项目。
- 元数据不完整时允许使用现有卡片兜底显示；不能因缺少封面、作者或时长丢弃合法记录。

## 测试策略

### 存储单元测试

- 合法记录的读写、倒序和 schema 归一化。
- 同一 `bvid` 重复写入时更新记录并移到最前。
- 不同分 P 的同一 `bvid` 仍只保留一条，并使用最新 `cid`。
- 超过 50 条时裁剪最旧记录。
- 缺少 `bvid` 或合法 `viewedAt` 的记录被丢弃。
- 损坏 JSON、未知结构及存储 API 异常返回空列表或静默失败。
- 新记录缺少元数据时保留旧记录中的有效展示字段。

### 合并逻辑单元测试

- 本地与 B 站记录按时间统一排序。
- 相同 `bvid` 跨来源只保留一项。
- 较新来源决定 `cid` 和进度，缺失字段由另一来源补全。
- B 站记录缺少可用时间时保持稳定顺序。
- 不同 `bvid` 即使标题相同也不得合并。

### 播放器与应用测试

- 投屏普通视频成功开始播放且有 `bvid` 时写入。
- 普通应用内播放、直播投屏、启动失败及缺少 `bvid` 时不写入。
- 同一播放会话的重复 `playing` 事件和周期性进度更新不得再次写入或刷新 `viewedAt`。
- 重新选择“最近观看”会刷新本地和 B 站数据。

### 页面测试

- 接口成功时展示合并后的去重列表。
- 未登录、接口错误和超时时，本地有记录则展示记录及非阻塞提示。
- B 站历史不可用且本地为空时保留当前阻塞状态。
- 本地损坏不影响 B 站历史展示。
- 两边均为空时显示现有空态。

## 验收标准

- 带 `bvid` 的普通视频投屏首次触发媒体 `playing` 状态后，无需等待 B 站心跳同步即可在刷新后的“最近观看”中找到该视频。
- 收到但未成功播放的投屏命令不会进入本地列表。
- 缺少 `bvid` 的投屏不会产生不可点击的最近观看卡片。
- 本地投屏记录最多保留 50 条，同一 `bvid` 只有一条。
- 同一视频同时存在于本地记录和 B 站历史时，页面只显示一张卡片，并按较新的观看时间定位。
- B 站历史接口仍会正常请求并参与合并，不被本地记录替代。
- 未登录或 B 站历史失败时，本地记录仍可查看，页面明确提示账号历史未加载。
- 本地数据损坏、配额不足或存储 API 异常不影响播放和 B 站历史。
- 重新选择“最近观看”后能看到会话期间新增的本地投屏记录。

## 风险与处理

- **来源时间不可完全对齐**：B 站历史时间由服务端提供，本地时间来自电视。合并按可用时间直接比较；电视系统时间异常可能影响排序，但不影响数据可用性。
- **元数据暂时不完整**：播放器成功开始时详情可能仍不完整。先保存可重播记录，合并时使用 B 站数据补全。
- **本地数据只属于当前设备**：清除应用数据、重装或更换电视会丢失，不宣称跨设备同步。
- **B 站历史同步延迟造成暂时重复来源**：按 `bvid` 合并，避免服务端记录稍后出现时展示两张卡片。
- **列表长度增加**：合并后最多 74 项；继续使用现有网格，不在本次增加分页或虚拟化。
