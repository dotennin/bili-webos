# 投屏标题与观看历史设计

## 背景

投屏播放普通视频后，播放器 overlay 应优先显示投屏命令携带的标题。由于 B 站发送端多数投屏命令不提供顶层 `title`，应用需要使用视频标识查询详情补齐真实标题；只有命令标题和详情标题都不可用时，才显示兜底文案“投屏视频”。同时，投屏播放的视频应像应用内直接播放的视频一样，正常向 B 站上报播放历史，使其进入账号的最近观看记录。

## 目标

- 投屏普通视频时，播放器 overlay 按 `command.title`、视频详情标题、“投屏视频”的顺序选择标题。
- 投屏普通视频时，继续走现有普通视频播放页的播放心跳/进度上报逻辑。
- 最近观看来源仍为 B 站历史接口，不新增本地投屏最近观看列表。

## 非目标

- 不为未登录或 B 站历史接口失败场景新增本地历史兜底。
- 不改变直播投屏标题逻辑，除非普通视频改动中需要保持同类兜底一致性。
- 不重构播放器、投屏订阅或历史页整体架构。

## 推荐方案

采用最小范围修复：

1. 投屏命令带有非空 `title` 时，播放器直接显示该标题，不发起仅用于补标题的额外请求。
2. 投屏命令没有标题时，不在 `App` 中立即写入“投屏视频”。应让 `PlayerPage` 识别“标题待解析”状态，并通过 `bvid` 或 `aid` 请求视频详情；该逻辑不能依赖 `cid` 是否存在。
3. 详情请求返回非空标题时，overlay 显示详情标题。
4. 详情请求失败、返回空标题或缺少可查询的视频标识时，overlay 最终显示“投屏视频”。
5. 投屏普通视频仍进入 `PlayerPage`，复用普通视频播放的心跳/进度上报路径，让 B 站服务端历史记录自然更新。

## 数据流

```text
投屏服务收到 play 命令
  -> App castSubscribe 回调构造 playerVideo
  -> PlayerPage 加载视频与详情
  -> 标题解析：command.title -> 视频详情标题 -> “投屏视频”
  -> 播放过程中执行现有 B 站心跳/进度上报
  -> B 站历史接口返回该视频
  -> HistoryPage 展示最近观看
```

## 标题显示规则

普通视频投屏标题优先级必须为：

1. `command.title`：投屏端已提供非空标题时直接使用。
2. `getVideoInfo` 返回的标题：投屏端未提供标题时，使用 `bvid` 或 `aid` 查询并补齐。
3. `投屏视频`：详情请求失败、详情标题为空，或投屏命令只有 `cid`/`epid` 而没有详情接口支持的视频标识时使用。

不能在 `App` 构造 `playerVideo` 时先把缺失标题替换为“投屏视频”，再由 `PlayerPage` 通过 `!video.title` 判断是否查询详情。这样 fallback 会被误判为真实标题，详情查询永远不会执行。`App` 应保留空标题或显式传递“标题缺失”状态，fallback 只能在详情解析完成或确认无法解析后应用。

当前代码中，`PlayerPage` 原有逻辑只在缺少 `cid` 时调用 `getVideoInfo`。实现需要增加与 `cid` 解耦的标题解析路径，在 `fromCast && !commandTitle && (bvid || aid)` 时请求详情。若缺少 `bvid` 和 `aid`，或请求没有得到非空标题，则设置最终 fallback“投屏视频”。

## 原因调查

标题数据流如下：

1. `nvaSession.decodeFrame` 从投屏帧中读取 action 和 JSON body，不转换 title 字段。
2. `CastController.handleCommand` 解析 `Play` 或 `PlayUrl` payload。
3. `normalizePlayPayload` 目前只从 `payload.title` 读取标题，并原样写入 intent。
4. `service.ts` 将 intent 原样发布给 `castSubscribe`。
5. `App.tsx` 从 `command.title` 构造 `playerVideo`。

仓库证据表明 service 没有在上述链路主动删除已有的顶层 `title`。但是 `cast-controller.test.ts` 仅使用人工构造且明确带有 `title` 的理想 payload，没有真实 B 站手机客户端投屏帧 fixture，也没有覆盖 title 缺失、空字符串或其他嵌套位置的场景。结合实机上 `command.title` 几乎始终为空的现象，当前最可信的根因是：B 站发送端多数投屏命令本身不发送顶层 `title`，现有解析器和测试错误地假设该字段通常存在。

实现阶段应增加边界诊断以验证该结论：在不记录完整标题和敏感 payload 的前提下，记录 action、payload 顶层 key、`nva_ext.content` key、视频标识类型以及 title 是否为非空字符串。若真实 payload 存在其他稳定标题字段，再把该字段加入 normalization；不能凭猜测添加字段别名。

## 最近观看规则

- 不新增本地存储项。
- 不修改 `HistoryPage` 的数据来源；它继续调用 B 站历史接口。
- 投屏普通视频必须复用普通视频播放器中的历史/心跳上报逻辑。
- 验收以 B 站账号历史记录是否出现该视频为准。

本地播放进度 `resume_progress` 与最近观看是两个不同概念。投屏视频可以继续参与现有播放进度恢复逻辑，但本次不新增“本地最近观看”列表，也不把 `resume_progress` 当作最近观看数据源。

## 已知限制

- 当前投屏普通视频对象的 `owner.name` 可能为空，因此 overlay 副标题可能不显示 UP 主信息。本次目标只要求标题正确，不要求补齐作者信息。
- 未登录账号无法写入 B 站历史记录；这是现有 B 站账号历史机制限制。
- 如果投屏命令只有 `cid` 或 `epid`，现有 `getVideoInfo` 无法直接查询标题，应立即使用 fallback“投屏视频”，不能阻塞播放。

## 验收标准

- 投屏命令包含 `title: "示例视频标题"` 时，播放器 overlay 显示“示例视频标题”，且不发起仅用于补标题的详情请求。
- 投屏命令不包含 `title`，但包含 `bvid` 或 `aid`，且视频详情接口返回标题时，overlay 最终显示详情标题。
- 投屏命令不包含 `title`，且视频详情接口失败或返回空标题时，overlay 最终显示“投屏视频”。
- 投屏命令不包含 `title`，且没有 `bvid`/`aid` 可用于详情查询时，overlay 显示“投屏视频”，播放流程不因标题解析而失败。
- 投屏播放普通视频一段时间后，B 站历史记录能看到该视频。
- 应用“最近观看”页刷新后，能通过 B 站历史接口展示该投屏播放记录。

## 测试建议

- service 单元测试：覆盖顶层 `title`、title 缺失、空字符串以及 `PlayUrl` 的 `nva_ext.content` payload，确认 normalization 不会丢失已有标题。
- service fixture 测试：在取得脱敏真实投屏帧后固定 action、payload 结构和标题存在性，避免继续依赖理想化 payload。
- 单元/渲染测试：模拟投屏 `play` 命令带标题，断言播放器显示命令标题，且不为补标题额外调用 `getVideoInfo`。
- 单元/渲染测试：模拟投屏 `play` 命令不带标题但带 `bvid`/`aid` 和 `cid`，视频详情返回标题，断言 overlay 最终更新为详情标题。
- 单元/渲染测试：模拟投屏命令不带标题且详情接口失败或返回空标题，断言 overlay 最终显示“投屏视频”。
- 单元/渲染测试：模拟投屏命令只有 `cid`/`epid`，断言不请求无法执行的详情接口、显示“投屏视频”且不阻断播放。
- 回归测试：确认投屏普通视频仍触发普通视频播放页已有的心跳/进度上报调用；自动化测试可优先验证心跳调用，手动测试再验证服务端历史最终出现。
- 手动验证：在已登录账号上投屏播放视频，等待心跳上报后刷新最近观看页。

## 风险与处理

- 如果投屏端不传标题且视频详情接口失败，overlay 显示“投屏视频”，确保用户始终能看到稳定标题。
- 如果 B 站服务端历史写入有延迟，最近观看页可能需要刷新或等待一段时间。
- 如果用户未登录，B 站历史不会更新；这是现有账号历史机制限制，不在本次范围内解决。
